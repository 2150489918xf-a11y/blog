# 管理后台 CMS 分层重构设计稿

## 一、背景

当前项目已经存在两套内容来源：

- 文章内容由 [admin.html](admin.html)、[supabase.js](supabase.js) 和 [db/migration.sql](db/migration.sql) 驱动，支持基础的登录、发布和编辑。
- 项目、资源导航、表格内容以及部分首页和简历页展示内容仍然直接写死在 [script.js](script.js) 与若干 HTML 页面中。

这导致管理后台只能管理文章，无法统一管理站点中的其他展示内容。

## 二、目标

- 将 Supabase 作为正式模式下的唯一内容源。
- 将后台从“文章编辑页”重构为“内容管理工作台”。
- 支持文章、项目、资源导航、表格内容、页面配置等内容的统一管理。
- 逐步让前台页面从静态数组和硬编码 HTML 过渡为数据库驱动。

## 三、范围边界

本次重构聚焦“内容管理系统”，不把项目扩展成完整用户系统。

纳入后台管理的内容：

- 文章
- 项目展示
- 资源导航分组与链接
- 表格展示内容
- 站点设置
- 导航配置
- 页面区块配置
- 简历页内容区块
- 媒体资源

不作为本次核心范围的内容：

- 登录页、注册页、设置页的业务化用户数据持久化
- 多角色权限系统
- 后端 API 服务拆分

## 四、架构决策

### ADR-001：采用“业务内容层 + 页面配置层”的分层模型

#### Context

- 文章、项目、资源链接和表格项具有天然的列表化和 CRUD 特征。
- 首页、导航、侧栏、简历等部分更像页面配置，不适合全部建成强类型业务表。

#### Decision

采用两层模型：

- 业务内容层：使用强类型表存储可排序、可筛选、可独立编辑的实体。
- 页面配置层：使用配置表存储导航、站点设置、页面区块和简历模块。

#### Trade-offs

- 放弃“单表万能 CMS”的快速扩展性。
- 换取更清晰的数据边界、更好的后台表单可维护性以及更易解释的答辩结构。

### ADR-002：继续保留静态前端形态，使用 Supabase 作为内容存储

#### Context

- 项目本体仍然是静态 HTML + CSS + JavaScript 站点。
- 现有文章系统已经基于 Supabase 工作，延续它比重新自建一套本地配置体系成本更低。

#### Decision

- 不引入框架化前端改造。
- 通过扩展 [supabase.js](supabase.js) 作为统一内容访问层。
- 通过更新 [script.js](script.js) 和管理端脚本，让前台页面逐步切换为数据库驱动。

#### Trade-offs

- 接受静态页面中部分 DOM 结构需要补充挂点。
- 接受短期内存在“数据库优先，静态内容兜底”的兼容期。

## 五、内容模型

### 业务内容层

- `articles`
- `article_categories`
- `projects`
- `resource_groups`
- `resource_links`
- `schedule_items`
- `media_assets`

### 页面配置层

- `site_settings`
- `navigation_items`
- `page_sections`
- `profile_blocks`

## 六、后台模块划分

- 内容总览：统计、最近更新、快速入口。
- 文章管理：文章列表、筛选、编辑、发布。
- 项目管理：项目卡片的增删改查。
- 资源管理：分组与链接的增删改查。
- 表格管理：表格展示内容的排序与状态维护。
- 页面配置：站点设置、导航、首页区块、简历区块。
- 媒体库：上传、复用和删除图片资源。

## 七、数据流设计

### 前台

- 页面不直接拼 Supabase 查询。
- 统一通过 [supabase.js](supabase.js) 暴露的内容访问方法取数。
- [script.js](script.js) 从“静态内容仓库”转为“远程优先 + 本地兜底的渲染层”。

### 后台

- 将 [admin.html](admin.html) 从内联脚本驱动重构为模块化管理台。
- 新增独立后台脚本，负责模块切换、列表渲染、表单渲染和保存逻辑。
- 复用统一内容访问层进行 CRUD。

## 八、迁移顺序

### 第一阶段

- 重构后台壳层和导航。
- 扩展数据库与 Supabase API。
- 接入文章、项目、资源、表格四个核心内容模块。

### 第二阶段

- 改造前台资源页、项目页、表格页为数据库驱动。
- 为首页和简历页补充页面配置挂点。

### 第三阶段

- 接入站点设置、导航配置、页面区块、简历区块。
- 完成媒体库和页面配置管理。

## 九、验证策略

- 每次修改后先检查相关文件语法和编辑器错误。
- 在浏览器中验证后台页面能够正常加载且文章模块不回归。
- 对项目、资源、表格等页面执行远程优先、静态兜底的展示验证。

## 十、风险与缓解

- 风险：静态 HTML 页面中存在较多硬编码内容。
  缓解：先为已有 JS 渲染区域接入远程数据，再逐步为静态区块补充挂点。

- 风险：正式模式与课程“纯静态提交”要求存在冲突。
  缓解：当前分支以正式模式为主，后续如需课程提交，可增加导出或静态快照方案。
